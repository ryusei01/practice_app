# エッジ（CDN / WAF）とアップロード制限

本番で API や Web を **Cloudflare** 等のリバースプロキシの背後に置く場合の推奨設定と、アプリケーション側の制限の関係をまとめる。

## 1. 信頼できるプロキシとクライアント IP

バックエンドは環境変数 **`TRUST_X_FORWARDED_FOR=true`** のとき、`X-Forwarded-For` の**先頭**をクライアント IP としてレート制限・OAuth 失敗カウントに使う。

- **前提:** プロキシがクライアント由来の `X-Forwarded-For` を上書き・正規化し、不正な多段ヘッダを付与されないこと。
- **本番（Render / Cloudflare 経由）**で IP 制限を正しく効かせる場合は `true` を推奨。
- ローカル直叩きでは `false` のまま。

## 2. WAF・レート制限（Cloudflare 例）

| 項目 | 目的 |
|------|------|
| **WAF マネージドルール** | OWASP 系シグネチャ、既知攻撃パターンのブロック |
| **レート制限ルール** | パス `/api/v1/auth/google` などログイン・決済起点への集中アクセスをエッジで抑制 |
| **Bot Fight / Bot Management** | 自動化トラフィックの検知（プランに応じて） |
| **国・IP アクセスルール** | 明らかに不要な地域・既知の悪性 IP の拒否 |

アプリ側の `slowapi` と**二重**になってよい（エッジで大流量を落とし、アプリでユーザー単位を制御する）。

## 3. リクエストボディサイズ

- **問題 CSV 一括アップロード:** アプリ側で **`BULK_CSV_MAX_BYTES`**（既定 5 MiB）を超えると `413`。
- **プロキシ / ロードバランサ**でも `client_max_body_size` 相当の上限を設定し、アプリより手前で拒否するとメモリ効率がよい。
