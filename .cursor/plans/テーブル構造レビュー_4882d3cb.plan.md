# テーブル構造レビューと改善提案

## 現在のテーブル一覧

`users`, `question_sets`, `questions`, `answers`, `user_question_stats`, `user_category_stats`, `purchases`, `reviews`, `otp_codes` の9テーブル。

---

## 問題1: users テーブルの肥大化と冗長性

**ファイル:** [backend/app/models/user.py](backend/app/models/user.py)

### 1-1. `role` と `is_seller` の役割整理

現在 `role`（Enum: USER/SELLER/ADMIN/SUPER_ADMIN）と `is_seller`（Boolean）が共存している。一見冗長だが、**seller が同時に学習者（問題を解く側）でもあるケース**を考慮すると、role を排他的 Enum にしたまま SELLER を含めるのは不適切。

seller は「アクセスレベル（role）」ではなく「追加の機能権限」であるため、以下のように整理すべき:

- `**role**`: アクセスレベルを表す（USER / ADMIN / SUPER_ADMIN の3値）
- `**is_seller**`: 販売機能の有効化フラグ（role とは独立）

これにより、以下の組み合わせが自然に表現できる:

- `role=USER, is_seller=False` — 一般学習者
- `role=USER, is_seller=True` — 学習者 兼 販売者（自分の問題集を売りつつ、他の問題集で学習）
- `role=ADMIN, is_seller=True` — 管理者 兼 販売者
- `role=ADMIN, is_seller=False` — 管理者（販売機能なし）

**提案:**

1. `UserRole` Enum から `SELLER` を削除する（USER / ADMIN / SUPER_ADMIN の3値にする）
2. `is_seller` を「後方互換性のため」ではなく、正式な販売機能フラグとして位置づける
3. 既存コードで `role == SELLER` を参照している箇所を `is_seller == True` に置換する
4. seller が他人の問題集を購入・学習できることを前提に、purchases / answers のロジックを確認する

### 1-2. OTP情報の二重管理

`users` テーブルに `otp_code`, `otp_expires_at` カラムがある一方、独立した `otp_codes` テーブルも存在する。2FAルーター（`two_factor.py`）は users テーブルの OTP カラムを使い、`core/otp.py` は `otp_codes` テーブルを使っている。

**提案:** `otp_codes` テーブルに統一する（複数OTP対応・purpose管理が可能）。`users` テーブルの `otp_code`, `otp_expires_at` カラムは削除する。

### 1-3. refresh_token が1つだけ

`refresh_token` が `users` テーブルに1カラムしかないため、複数デバイスからの同時ログインができない（後からログインした端末のトークンで上書きされる）。

**提案:** 別テーブル `user_sessions` を作成するか、現状の1デバイス制限を仕様として受け入れるか判断が必要。

### 1-4. 2FA関連カラムの未使用

`two_factor_secret`（TOTPシークレット）が「将来の拡張用」として存在するが使われていない。`backup_codes` はJSON文字列として1カラムに格納されている。

**提案:** 未使用カラムは削除するか、実装予定が明確なら残す。backup_codes のJSON格納は現段階では許容範囲。

## 問題2: ユニーク制約の欠如

### 2-1. user_question_stats に (user_id, question_id) のユニーク制約がない

同じユーザー×問題の組み合わせで複数行が作成されるリスクがある。API側のupsertロジックに依存しているが、DBレベルでの保証がない。

### 2-2. user_category_stats に (user_id, category) のユニーク制約がない

同上。

### 2-3. purchases に (buyer_id, question_set_id) のユニーク制約がない

同じ問題集を二重購入できてしまう可能性がある。

### 2-4. reviews に (user_id, question_set_id) のユニーク制約がない

同じユーザーが同じ問題集に複数レビューを投稿できてしまう。

**提案:** 上記4テーブルすべてに複合ユニーク制約を追加する。

```python
# user_question_stats
__table_args__ = (UniqueConstraint('user_id', 'question_id', name='uq_user_question'),)

# user_category_stats
__table_args__ = (UniqueConstraint('user_id', 'category', name='uq_user_category'),)

# purchases
__table_args__ = (UniqueConstraint('buyer_id', 'question_set_id', name='uq_buyer_question_set'),)

# reviews
__table_args__ = (UniqueConstraint('user_id', 'question_set_id', name='uq_user_review'),)
```

## 問題3: 非正規化された集計カラムの整合性リスク

### 3-1. question_sets の集計カラム

- `total_questions` — questions のCOUNTと乖離する可能性
- `total_purchases` — purchases のCOUNTと乖離する可能性
- `average_rating` — reviews の平均と乖離する可能性
- `average_difficulty` — questions の平均と乖離する可能性

### 3-2. questions の集計カラム

- `total_attempts`, `correct_count`, `average_time_sec` — answers から計算されるべきだが、`StatsUpdater` のみで更新される

**提案:** 非正規化自体はパフォーマンスのために合理的だが、以下のいずれかが必要:

- (A) DBトリガーで自動同期する
- (B) 定期バッチで再計算する（`recalculate-stats` APIは存在するがユーザー統計のみ）
- (C) 現状のアプリ層更新を維持しつつ、整合性チェックAPIを追加する

## 問題4: 外部キーの ON DELETE 未指定

`otp_codes` のみ `ondelete="CASCADE"` が設定されているが、他のテーブルのFKにはON DELETE指定がない。


| FK                                           | 現状                  | リスク          |
| -------------------------------------------- | ------------------- | ------------ |
| `question_sets.creator_id` -> users          | 指定なし                | ユーザー削除時にFK違反 |
| `questions.question_set_id` -> question_sets | 指定なし（ORM cascadeあり） | SQLレベルでは保護なし |
| `answers.user_id` -> users                   | 指定なし                | ユーザー削除時にFK違反 |
| `answers.question_id` -> questions           | 指定なし                | 問題削除時にFK違反   |
| `purchases.buyer_id` -> users                | 指定なし                | ユーザー削除時にFK違反 |
| `reviews.user_id` -> users                   | 指定なし                | ユーザー削除時にFK違反 |


**提案:** 各テーブルの業務要件に応じて `ondelete` を設定する:

- `CASCADE`: 親削除時に子も削除（例: questions -> question_sets）
- `SET NULL`: 親削除時にnullにする（例: purchases -> users は購入履歴保持のためSET NULL）
- `RESTRICT`: 参照されている限り親の削除を禁止（最も安全）

## 問題5: reviews テーブルのリレーション未定義

[backend/app/models/marketplace.py](backend/app/models/marketplace.py) の `Review` モデルにはFKがあるが、`relationship()` が定義されていない。また、API側にレビュー用のエンドポイントが一切存在しない。

**提案:** レビュー機能を実装予定なら relationship を追加する。不要なら `reviews` テーブル自体を削除する。

## 問題6: 主キーの型

全テーブルで `Column(String, primary_key=True)` を使用。アプリ側で UUID を文字列として生成していると推測される。

**リスク:**

- UUID文字列（36文字）はインデックスサイズが整数型の約9倍
- JOIN/FK参照時の比較コストが高い

**提案:** 現段階では大きな問題にはならないが、データ量増加時にパフォーマンスに影響する可能性がある。将来的には PostgreSQL の `UUID` 型の使用を検討する。今すぐの変更は不要。

---

## 問題7: インデックス不足

頻出クエリパターンに対してインデックスが不足している:

- `answers` テーブル: `(user_id, question_id)` の複合インデックスがない（推薦クエリで頻用）
- `answers` テーブル: `(user_id, answered_at)` の複合インデックスがない（履歴クエリで頻用）
- `questions` テーブル: `(question_set_id, order)` の複合インデックスがない（順序取得で頻用）
- `reviews` テーブル: `user_id` に `index=True` がない

---

## 問題8: docs/sql/ とモデル定義の乖離

`docs/sql/database_setup.sql` 等のスキーマでは `question_sets.user_id`, `question_sets.is_public` を使っているが、実際のSQLAlchemyモデルでは `creator_id`, `is_published` を使用。RLSポリシーの一部（`rls_policies_only.sql`）も古いカラム名を参照している。

**提案:** 古いSQLファイルを現行モデルに合わせて更新するか、廃止マーキングする。

## 優先度まとめ


| 優先度 | 項目                             | 影響              |
| --- | ------------------------------ | --------------- |
| 高   | ユニーク制約の追加（問題2）                 | データ不整合の直接原因     |
| 高   | ON DELETE の設定（問題4）             | FK違反によるエラー      |
| 中   | OTP二重管理の解消（問題1-2）              | コード混乱・保守性       |
| 中   | role/is_seller 役割整理（問題1-1）     | SELLER兼学習者の表現   |
| 中   | reviews のリレーション/API（問題5）       | 使われていないテーブル     |
| 低   | 複合インデックス追加（問題7）                | パフォーマンス（データ量次第） |
| 低   | 集計カラムの整合性対策（問題3）               | 表示値の不一致         |
| 低   | docs/sql の整理（問題8）              | ドキュメントの正確性      |
| 情報  | 主キー型（問題6）                      | 将来のスケーラビリティ     |
| 判断  | refresh_token / マルチデバイス（問題1-3） | 仕様次第            |
